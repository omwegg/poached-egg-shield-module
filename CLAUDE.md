# CLAUDE.md

このファイルは、Claude Code（および他の AI エージェント）がこのリポジトリで作業する際のガイドです。
人間向けの概要は [README.md](README.md) を参照してください。

## このリポジトリの正体

自作分割キーボード **Poached Eggs** 用の **ZMK 外部モジュール 兼 ユーザー設定** リポジトリです。
アプリケーションコードではなく、**Zephyr/ZMK の devicetree・Kconfig 設定の集合体**である点に注意してください。「ビルド」とは GitHub Actions 上で ZMK 本体を取得してファームウェア `.uf2` を生成することを指します。

- MCU: Seeeduino XIAO BLE (`seeeduino_xiao_ble`)
- 左右分割（左=セントラル / 右=ペリフェラル）
- マトリクス: 4 行 × 14 列、`col2row`
- ZMK 本体は **v0.2.1 に固定**

## ファイルの役割と編集時の勘所

| ファイル | 役割 | 編集時の注意 |
|----------|------|--------------|
| `config/poached_eggs.keymap` | 各レイヤーのキー定義 | 各行のバインディング数が**マトリクス変換の map と一致**している必要がある |
| `config/boards/shields/poached_eggs/poached_eggs.dtsi` | 物理レイアウト・matrix-transform・kscan | `columns`/`rows`、`map` の `RC(行,列)`、行 GPIO を定義 |
| `*_left.overlay` / `*_right.overlay` | 左右の列 GPIO | 右手は `&default_transform { col-offset = <7>; }` で列をずらしている |
| `config/poached_eggs.json` | ZMK Studio / keymap-editor 用レイアウト | **物理レイアウトを変えたら dtsi と JSON の両方**を同期させる |
| `build.yaml` | CI のビルド対象 | `shield` 名は `Kconfig.shield` の定義と一致させる。左のみ `studio-rpc-usb-uart` snippet を付与 |
| `Kconfig.defconfig` | 分割・セントラル役割など | 左 = `ZMK_SPLIT_ROLE_CENTRAL` |
| `*.conf` | `CONFIG_*` の有効化 | 共通は `poached_eggs.conf`、左固有（Studio 等）は `poached_eggs_left.conf` |

## 重要な不変条件（壊しやすい箇所）

1. **ZMK バージョンの同期** — `config/west.yml` の `revision` と `.github/workflows/build.yml` の `@v0.2.1` は**同じバージョン**を指す必要があります。上げ下げするときは両方を必ず合わせてください。
2. **キー数の一致** — `poached_eggs.keymap` の各レイヤーは合計 50 キー（左右で 6+6+6+7）。`dtsi` の物理レイアウト・`map`・`json` とズレないこと。
3. **左右の非対称設定** — セントラルは左、Studio snippet も左のみ、右は `col-offset=7`。左右を編集するときは対称性の前提を疑うこと。
4. **物理レイアウト変更は 3 ファイル連動** — `poached_eggs.dtsi`（`keys` と `default_transform`）、`poached_eggs.json`、必要なら `build.yaml`。

# 作業の基本ルール

## 言語

- **ユーザーとのやり取り・コミットメッセージ・サブエージェントへの指示・スキルの利用と出力は、原則すべて日本語**で行う。
- コード内コメントも日本語でよい（既存に合わせる）。技術用語・識別子・コマンドはそのまま英語で構わない。

## ドキュメントの参照と更新

- 作業を始める前に [README.md](README.md) と本 CLAUDE.md を**必ず参照**し、現状のルール・構成・不変条件を把握してから着手する。
- ファイル構成・キーマップ・物理レイアウト・ビルド手順・運用ルールなどを変更したら、**同じ作業の中で対応するドキュメント（README.md / CLAUDE.md）も更新**し、コードとドキュメントを乖離させない。
- ルールやワークフローに関する新しい取り決め（セッション中の決定など）は、忘れる前に CLAUDE.md へ反映する。

## 作業範囲

- 作業は原則**このリポジトリ（`poached-egg-shield-module`）内で完結**させる。親ディレクトリやホーム配下の他の場所、外部リポジトリを変更しない。
- ZMK 本体や Zephyr など、west が取得する外部の依存ソースは改変しない。
- サブエージェントに委譲する場合も、同じ作業範囲・同じルールを必ず引き継がせる。

## 確認の粒度と編集権限

リポジトリ内のファイルは、変更の影響度に応じて 4 段階で扱う。判断基準は「**元に戻しやすいか**」と「**デバイスの挙動・ビルド結果に影響するか**」。

| 段階 | 対象 | 進め方 |
|------|------|--------|
| ① 無確認で実行 | README/CLAUDE 等のドキュメント、コード内コメント、`.gitignore`、明白な typo・整形 | そのまま編集してよい |
| ② 軽い宣言のみ | `*.conf`、`build.yaml`、`Kconfig.*`、`*.zmk.yml` などの設定・メタデータ | 何をなぜ変えるか一言述べてから編集 |
| ③ 要事前確認 | `*.keymap`（＝キー挙動の変更）、`*.dtsi`/`*.overlay`（GPIO・マトリクス・物理レイアウト）、`west.yml`/`build.yml` のバージョン、ファイル削除・大規模リネーム | 変更内容を提示し、承認を得てから実行 |
| ④ 原則禁止（明示指示が必須） | リモートへの push / PR 作成、外部サービスへの送信、リポジトリ外のファイル変更、ZMK 本体ソースの改変 | ユーザーの明示的な指示があるときのみ |

- 承認は**意思決定が必要なものに絞る**。些末な選択は妥当な既定値で進め、事後に一言添える。
- 迷ったら一段安全側に倒す（②→③ のように上の段階として扱う）。

- 既存のスタイル・命名・コメント密度・インデントに合わせ、周囲のコードと馴染ませる。

- 上記「重要な不変条件」のチェックを毎回行う。
- **検証できていないことを「動く」と言わない。** 未検証の点は明示する。
- 最終的なビルド可否は GitHub Actions で確認する（次節参照）。

## ビルド確認の方針

**基本は GitHub Actions（リモート CI）で確認する。** 理由:

- `.github/workflows/build.yml` で既に構成済みで、追加の環境構築が不要。
- 実際に配布する `.uf2` と同じ成果物が得られ、ローカル環境差異の影響を受けない。
- ZMK 公式が推奨する user-config のワークフロー。

ただし Actions はトリガに push（または Actions タブの `workflow_dispatch`）が必要なため、**ビルド確認はリモートへの push を伴う**。「不要なものをリモートに上げない」ために次を守る:

- `main` へ直接 push しない。**検証用ブランチ**に push し、確認後はそのブランチを削除する。
- push 自体は**ユーザーの明示的な許可を得てから**行う（ビルド確認目的でも勝手に push しない）。
- ビルド成果物（`.uf2`/`.hex`/`.bin`）や west ワークスペースはコミットしない（`.gitignore` 済）。
- Actions の Artifacts は GitHub 側で自動失効するため、リポジトリ本体は汚れない。

**ローカルビルド環境は任意**（頻繁に反復する場合のみ検討）。採用する場合:

- west ワークスペースは**このリポジトリの外**に作り、依存物・成果物をリポジトリ内に持ち込まない。
- これによりリモートへ push せずにビルド検証ができる。

→ 結論: まず **GitHub Actions を基本**とし、push は都度ユーザー確認 + 検証ブランチ運用でリモートを汚さない。ビルド反復が増えてきたらローカル環境を追加検討する。

## コミットのルール

- **コミットメッセージは原則日本語**。`feat:`/`fix:`/`docs:`/`chore:`/`refactor:` などの接頭辞は英語でよい（既存履歴に合わせる）。
- **`main` 以外のローカルブランチでは、確認なしでコミットしてよい**（都度の許可は不要）。ただし `main` へは直接コミットせず、必要ならブランチを切る。
- **push はこの許可に含まれない。** リモートへの push / PR 作成は引き続き、ユーザーの明示的な指示があるときのみ行う（tier ④）。
- 1 コミット = 1 つの論理的変更。
- AI が作成したコミットは `Co-Authored-By` トレーラーで作成者を明示する。

## サブエージェント・スキルの利用ルール

- **広範な読み取り調査はサブエージェントを積極的に検討する**: ログや大きなファイル群の探索など「結論だけ受け取れば済む」調査は、Explore など読み取り専用エージェントに委譲してコンテキスト/トークンを節約する。
- 一方、**ファイルの編集・実行を伴う作業をサブエージェントに任せるのは、ユーザーが明示的に依頼したときのみ**。通常の変更は自分のツールで処理する。
- サブエージェントを使う場合も、本 CLAUDE.md のルール（作業範囲・編集権限・push 禁止・日本語）を**プロンプトに明記して引き継がせる**。
- スキルの利用・出力も日本語を基本とする。
